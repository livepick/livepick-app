# Lightweight Implementation

Implement small, pattern-following changes directly in the main context. No agent delegation — this is the fast path for contained work.

## Why no agent delegation

Spinning up a subagent adds overhead (context loading, rule reading, review delegation). For changes under 7 files that follow existing patterns, doing it directly is faster and produces the same quality because the patterns are already established.

## When this skill is the right choice

- Adding 1-2 API endpoints to an existing service + connecting them in the UI
- Adding a field through the full stack (model → repo → service → handler → frontend)
- Small UI additions (new card section, button, filter option)
- Bug-adjacent improvements within existing architecture

## When to escalate (not optional — lightweight skills shouldn't make architectural decisions)

| Signal | Escalate to |
|--------|------------|
| New domain package (`internal/<new>/`) needed | `/impl-backend-go` |
| New DB table or schema migration | `/impl-backend-go` |
| Structural design decision needed | `/design-feature` first |
| Total files will exceed 10 | `/impl-fullstack` or domain-specific |

## Workflow

### STEP 1: Scope Analysis

1. Extract key symbols from the request, search with Grep/Glob
2. Identify the **pattern to replicate** — find an existing file that does something similar
3. Read the relevant `.claude/rules/` files (only the ones that apply to this change)
4. Check escalation criteria — if any match, stop and suggest the right skill

### STEP 2: Design Direction (UI changes only)

When the change involves UI, take 30 seconds to think before coding. Read `.claude/rules/frontend-design.md` and consider:

- **Purpose**: What does this UI element help the user do?
- **Tone**: Does it match the page's existing feel? (active event = energetic, history = calm)
- **State visualization**: If showing event status, follow the status color/badge conventions

This step prevents the default "generic card with shadow and rounded corners" output. The goal is UI that feels intentionally designed for LivePick, not generated by AI.

### STEP 3: Change Plan → User Approval

Present a table of files to change/create, what pattern each follows, and the implementation order. Get approval before writing any code — this prevents wasted work if the user has a different mental model.

```markdown
## 변경 계획

| # | 영역 | 파일 | 변경 유형 | 설명 |
|---|------|------|----------|------|
| 1 | BE | path/to/file | Modify | 변경 내용 |
| 2 | FE | path/to/file | New | 변경 내용 |

### 복제 대상 패턴
- Backend: [기존 파일] — 이 패턴을 따름
- Frontend: [기존 파일] — 이 패턴을 따름
```

### STEP 4: Implement

Follow the approved plan. Order: Backend (model → repo → service → handler) → Frontend (API hook → component → page).

The core principle is **replicate existing patterns + follow the rules**. Don't introduce new structural patterns — that's what the larger skills are for.

For UI work specifically:
- Use Tailwind classes, not inline styles — consistency with the rest of the app matters more than one-off perfection
- Status badges and event states should follow the conventions in `frontend-design.md` section VII
- Typography hierarchy should be clear (title/body/caption — not five different font sizes)
- Motion should be purposeful (hover feedback on clickable items, not decorative spinners)

### STEP 5: Quick Review + Report

Self-check against the rules that apply:

**Backend checks:**
- Handler → Service → Repository layer separation maintained
- Error codes use AppError constants (no hardcoded strings)
- SQL uses parameterized queries
- No sensitive data in logs

**Frontend checks:**
- Data fetching via TanStack Query (not useEffect + useState)
- Tailwind classes (not `style={{}}`)
- Type safety — no `any`, error codes via constants
- Clickable elements have appropriate feedback (hover/active states)
- Event status uses the project's visual conventions

**Report**: what changed, build status, any notes worth highlighting.
