# Frontend Design & Aesthetics

> LivePick은 인플루언서 이벤트 플랫폼이다. 시각적 품질이 사용자 신뢰와 참여에 직접 영향을 미친다.
> 이 규칙은 UI 구현 시 코딩 전에 읽고, 디자인 결정의 근거로 사용한다.

---

## I. Design Thinking (코딩 전 필수)

UI 컴포넌트나 페이지를 구현하기 전에 다음을 먼저 결정한다:

| 질문 | 왜 중요한가 |
|------|-----------|
| **Purpose** — 이 화면이 해결하는 문제는? 누가 쓰는가? | 목적이 불명확하면 UI가 산만해진다 |
| **Tone** — 이 화면의 분위기는? (활기찬/차분한/신뢰감/재미) | 톤이 없으면 generic한 결과물이 나온다 |
| **Differentiation** — 사용자가 기억할 한 가지는? | 기억에 남지 않는 UI는 재방문을 만들지 못한다 |

LivePick의 기본 톤: **활기찬 + 신뢰감**. 인플루언서 이벤트의 흥미와 공정한 추첨의 신뢰를 동시에 전달.

---

## II. "AI Slop" 회피

AI가 생성하는 코드는 특정 패턴에 수렴하는 경향이 있다. 이를 의식적으로 피한다:

| 피해야 할 패턴 | 왜 피하는가 | 대안 |
|---------------|-----------|------|
| 보라색 그라데이션 + 흰 배경 | 가장 흔한 AI 생성 미학 | 프로젝트 브랜드 컬러 팔레트 사용 |
| Inter, Roboto, Arial 폰트만 사용 | 모든 AI 결과물이 똑같아 보인다 | 프로젝트 폰트 시스템 정의 후 사용 |
| 모든 것이 center 정렬 + 균일 rounded | 개성 없는 레이아웃 | 비대칭, 그리드 변형, 의도적 여백 |
| 모든 카드가 동일한 그림자 + 패딩 | 시각적 위계가 없다 | 중요도에 따라 차등 강조 |

핵심: 모든 디자인 선택에 **의도**가 있어야 한다. "기본값이니까"가 아니라 "이 화면에서 이것이 효과적이니까".

---

## III. Typography

- 프로젝트 폰트 시스템이 정의되면 그것을 따른다
- 숫자 데이터(참여자 수, 날짜, 카운트)는 tabular-nums 또는 monospace 계열로 가독성 확보
- 텍스트 위계: 화면당 최대 3~4단계 (title / subtitle / body / caption)
- 긴 텍스트는 `max-width: 65ch` 로 가독성 확보

---

## IV. Color & Theme

- 브랜드 컬러가 정의되면 CSS 변수로 관리
- 주요 색상 1개 + 보조 색상 1~2개 + neutral 계열로 구성
- 상태 표현: 성공(green), 경고(amber), 에러(red), 정보(blue) 시맨틱 컬러 사용
- 다크 모드는 추후 지원 시 CSS 변수 기반으로 전환 용이하게 설계

---

## V. Motion & Interaction

- 목적 있는 애니메이션만 사용 — 장식적 애니메이션은 로딩 시간을 늘린다
- 페이지 진입 시 staggered reveal (순차 등장)은 콘텐츠 목록에 효과적
- hover/press 피드백은 인터랙티브 요소에 필수 (클릭 가능함을 시각적으로 전달)
- `prefers-reduced-motion` 미디어 쿼리 대응
- 애니메이션 duration: 150~300ms (체감상 즉각적이면서 부드러운 범위)

## VI. Spacing System (여백 디자인 가이드)

> 모든 여백은 4px 기반 스케일을 사용한다. 일관된 수직 리듬이 전문적인 느낌을 만든다.
> **원칙: 모바일 우선. 작은 화면에서 여백이 과도하면 콘텐츠가 밀려 스크롤이 길어진다.**

### 1. Spacing Scale (Tailwind 토큰)

| 토큰 | 값 | 용도 |
|------|-----|------|
| `1` | 4px | 인라인 요소 간 미세 간격 |
| `2` | 8px | 텍스트 줄 간, 라벨↔값 |
| `3` | 12px | 아이콘↔텍스트, 밀접한 요소 그룹 |
| `4` | 16px | 카드 내부 패딩, 소항목 간 |
| `6` | 24px | **페이지 타이틀 subtitle ↔ 첫 번째 콘텐츠** |
| `8` | 32px | **섹션 간 구분** (같은 맥락) |
| `10` | 40px | 섹션 간 구분 (다른 맥락) |
| `12` | 48px | 사용 자제 — 매우 큰 구분이 필요한 경우만 |

### 2. 페이지 레이아웃 여백

```
TopNav (fixed, h-16)
│
├─ pt-20 (80px) ← 네비게이션 아래 콘텐츠 시작 (TopNav h-16 + 여유 16px)
│
├─ 페이지 타이틀 (h1)
│   mb-1 ~ mb-2 ← 타이틀 → 서브타이틀
│
├─ 서브타이틀/설명 (p)
│   mb-6 ← 헤더 영역 → 첫 번째 콘텐츠 섹션
│
├─ 필터/탭 바 (있는 경우)
│   mb-6 ← 필터 → 콘텐츠
│
├─ 콘텐츠 섹션
│   mb-8 ← 섹션 간
│
├─ pb-24 (96px) ← BottomNav 공간 확보
│
BottomNav (fixed, h-20)
```

### 3. 핵심 규칙

| 위치 | Tailwind 클래스 | px | 설명 |
|------|-----------------|-----|------|
| 페이지 상단 패딩 | `pt-20` | 80px | TopNav(64px) + 여유(16px) |
| 페이지 좌우 패딩 | `px-4 md:px-6` | 16/24px | 모바일은 16px로 콘텐츠 영역 확보 |
| 타이틀 → 서브타이틀 | `mb-1` | 4px | 밀접한 텍스트 그룹 |
| 헤더 영역 → 콘텐츠 | `mb-6` | 24px | **가장 중요한 간격** — 과도하면 안 됨 |
| 필터/탭 → 콘텐츠 | `mb-6` | 24px | 필터와 결과 사이 |
| 섹션 → 섹션 | `mb-8` | 32px | 독립적 섹션 구분 |
| 카드 그리드 gap | `gap-4 md:gap-6` | 16/24px | 카드 간 간격 |
| 카드 내부 패딩 | `p-4 md:p-6` | 16/24px | 카드 콘텐츠 여백 |
| 페이지 하단 패딩 | `pb-24` | 96px | BottomNav(80px) + 여유(16px) |

### 4. 금지 패턴

| 금지 | 이유 | 대안 |
|------|------|------|
| 헤더에 `mb-10` / `mb-12` | 타이틀↔콘텐츠 간 과도한 빈 공간 | `mb-6` (24px) |
| h1에 `mb-8` | 타이틀 아래 불필요한 공백 | `mb-1` ~ `mb-2` (서브타이틀까지만) |
| `pt-24` (96px) 페이지 상단 | TopNav 높이 대비 과도 | `pt-20` (80px) |
| 페이지마다 다른 상단 패딩 | 화면 전환 시 점프감 | 모든 페이지 동일한 `pt-20` |
| `pb-32` (128px) 하단 | BottomNav 대비 과도 | `pb-24` (96px) |

### 5. Spatial Composition (기존)

- 카드 그리드: 모바일 2열 → 데스크탑 최대 6열
- 시각적 위계: 크기, 색상, 여백으로 중요도 차이를 만든다
- 여백은 콘텐츠를 "숨 쉬게" 한다 — 단, **과도한 여백은 콘텐츠 밀도를 떨어뜨린다**
- 섹션 구분: 배경색 변화 또는 여백으로 (불필요한 구분선 최소화)

## VII. 이벤트 상태 시각화

LivePick의 핵심 정보인 이벤트 상태를 명확하게 전달한다:

| 상태 | 시각적 표현 |
|------|-----------|
| 예정 (pending) | 차분한 톤, "D-N" 카운트다운 |
| 진행 중 (active) | 강조 색상, 참여 CTA 버튼, 참여자 수 실시간 |
| 마감 (closed) | 흐린 톤, "마감" 배지, 결과 확인 유도 |
| 당첨 (won) | 축하 표현, 강조 카드 |
